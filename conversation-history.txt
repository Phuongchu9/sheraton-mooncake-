This is a **REST API** using JSON. Here's the full breakdown:

---

## API Type: REST (JSON over HTTP)

**Authentication:** Custom token header `x-admin-token` (JWT or similar, returned after login).

---

## Endpoints

| Method | Endpoint | Auth | Description |
|--------|----------|------|-------------|
| `POST` | `/api/admin/login` | None | Login, returns a token |
| `GET` | `/api/orders` | `x-admin-token` | List orders with filtering |
| `PATCH` | `/api/orders/:id` | `x-admin-token` | Update order status |
| `DELETE` | `/api/orders/:id` | `x-admin-token` | Delete an order |

---

## Details

**`POST /api/admin/login`**
```json
// Request body
{ "password": "..." }

// Response
{ "token": "..." }   // or { "error": "..." }
```

**`GET /api/orders?status=all&search=...`**
```json
// Response
{
  "orders": [{ "id", "name", "phone", "email", "product", "quantity", "message", "status", "created_at" }],
  "stats": { "total", "new_count", "contacted_count", "completed_count" }
}
```

**`PATCH /api/orders/:id`**
```json
// Request body
{ "status": "new" | "contacted" | "completed" | "cancelled" }
```

**`DELETE /api/orders/:id`** — no body, returns 200 on success.

---

**Summary:** Plain REST API, JSON request/response bodies, token auth via a custom `x-admin-token` header. No GraphQL, no gRPC, no WebSockets.

**No PostgreSQL** — it's not using any database at all.

The "database" is a plain **JSON file** (`orders.json`) on disk:

- **Storage:** `orders.json` in the project root
- **No DB driver** — zero database dependencies (`pg`, `mysql2`, `mongoose`, etc. are absent from `package.json`)
- **How it works:** Every read loads the entire file; every write serializes the whole array back to disk with `fs.writeFileSync`

```
sheraton-mooncake/
└── orders.json   ← this IS the database
```

This is a simple flat-file approach — fine for low traffic, but it has no concurrency protection (concurrent writes could corrupt the file) and will get slow as `orders.json` grows. If you want to upgrade, SQLite (`better-sqlite3`) would be the easiest drop-in, or PostgreSQL if you need multi-instance/production scale.

- **Thiếu meta description** (`index.html:1-8` — không có `<meta name="description">`). Google/AI cần đoạn mô tả ngắn 150-160 ký tự.
- **Thiếu Open Graph tags** — khi chia sẻ link, không có ảnh preview, không có title/description cho mạng xã hội.
- **Thiếu Schema.org structured data.** Không có `Product`, `LocalBusiness`, hay `FAQPage` schema. AI như Perplexity, Google AI Overview ưu tiên trang có schema.
- **Title tag quá ngắn:** `Bánh Trung Thu Sheraton 2026` — thiếu từ khóa địa lý "Hà Nội", thương hiệu "Sheraton Hanoi Hotel", và từ khóa người dùng tìm kiếm.

### 4. Performance

**Vấn đề nghiêm trọng nhất:**

- **Ảnh hero quá lớn.** 10 file JPG/PNG x 14-32MB = tổng ~200MB cho slideshow. Trên 4G Việt Nam (trung bình 20Mbps), ảnh đầu tiên mất 5-13 giây. Cần export lại: max 1920px, JPEG quality 75%, mỗi ảnh < 300KB.
- **Google Fonts render-blocking** (`index.html:8`). `<link rel="stylesheet">` sync trong `<head>` block render. Thêm `media="print" onload="this.media='all'"` hoặc dùng `font-display: swap`.
- **Hero JS load tất cả 10 ảnh ngay lập tức.** Không có lazy loading cho slideshow — browser prefetch tất cả background-image. Chỉ nên load ảnh kế tiếp khi slide gần hiện.

**Vấn đề vừa:**

- `db.js` đọc toàn bộ `orders.json` cho mỗi request (`getAll`, `getStats` đều gọi `readDB()`). Khi có 1000+ đơn, file parse sẽ chậm và tốn memory. Hiện tại chấp nhận được nhưng cần chú ý.
- Không có HTTP cache headers cho ảnh sản phẩm trong `server.js` — browser re-download ảnh mỗi lần.

### 5. Code Quality

**Bug tiềm ẩn:**

- **Admin token = plain password** (`server.js:65`, `server.js:96`). `res.json({ token: ADMIN_PASSWORD })` trả về mật khẩu dưới dạng token. Ai inspect Network tab thấy ngay mật khẩu. Nên dùng JWT hoặc session token ngẫu nhiên.
- **Không có rate limiting** trên `POST /api/orders` (`server.js:53`). Bot có thể spam hàng nghìn đơn giả, làm đầy `orders.json`.
- **XSS tiềm ẩn trong admin.html.** Nếu admin render `order.name`, `order.message` trực tiếp vào innerHTML mà không escape, khách độc hại có thể nhúng `<script>` vào field tên/ghi chú → chạy trong trình duyệt của admin.
- **`cors()` không giới hạn origin** (`server.js:12`). Bất kỳ domain nào cũng có thể gọi API. Nên chỉ allow `https://sheraton-mooncake.onrender.com`.
- **Cart dùng `sessionStorage`** — mất khi đóng tab. Dùng `localStorage` thì giỏ hàng giữ lại giữa các phiên.

**Logic chưa tối ưu:**

- `quantity` trong form là free-text input, không có validation server-side (`server.js:58` chỉ validate phone). Khách có thể submit `quantity: -1` hay `quantity: "abc"`.
- `openModal()` nhận argument product name nhưng nếu gọi từ cart panel sẽ không auto-populate đúng. Cart và modal dùng 2 luồng riêng biệt không sync.
